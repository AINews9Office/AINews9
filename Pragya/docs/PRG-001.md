# Executive Summary

Pragya is the AI Learning Companion for AINews9.

Unlike a generic conversational AI, Pragya is purpose-built to improve AI literacy by guiding users through structured learning journeys.

Pragya combines conversational AI with curated educational content, recommendations, and progressive learning paths.

Its objective is not merely to answer questions, but to help users continuously learn, confidently use, safely adopt, and professionally grow with Artificial Intelligence.

---

# Vision

Build India's most trusted AI Learning Companion that empowers every individual to confidently learn, use, and benefit from Artificial Intelligence.

---

# Mission

Make Artificial Intelligence simple, useful, safe, and accessible for every Indian through guided learning experiences powered by AI.

---

# Product Philosophy

Pragya is not designed to replace search engines or function as a general-purpose chatbot.

Pragya exists to help users understand AI through structured guidance, contextual conversations, curated knowledge, and personalized recommendations.

Every interaction should move the learner one step forward in their AI journey.

---

# Guided AI Learning Philosophy

Traditional chatbots begin with an empty conversation.

Pragya begins with a learning journey.

Instead of asking users to decide what to ask first, Pragya provides a structured onboarding experience that establishes learning context before the conversation begins.

This approach enables:

- Faster personalization
- More accurate responses
- Better recommendations
- Reduced ambiguity
- Progressive learning

Learning is viewed as a journey rather than a collection of isolated questions.

---

Pragya supports the four strategic pillars of AINews9.

- Learn AI
- Use AI
- Stay Safe
- Grow with AI

Every feature within Pragya should support one or more of these pillars.

# Part 2
# Product Requirements & Guided User Experience

---

# Overview

Pragya is designed around a Guided Learning Journey rather than an open-ended chat experience.

Instead of expecting users to formulate the perfect question, Pragya first establishes learning context and then provides personalized guidance throughout the conversation.

This approach enables more accurate responses, better recommendations, and a structured AI learning experience.

---

# Product Decision PD-001

## Guided Learning Journey

**Status:** APPROVED

Pragya shall always begin with a Guided Learning Journey before the first conversation starts.

The Guided Journey establishes the user's initial learning context.

Once established, the conversation becomes highly personalized.

---

# User Journey

Every new conversation should follow this flow.

```

Welcome

↓

Choose AI Journey

↓

Choose Persona

↓

Choose Topic

↓

Conversation

↓

Continue Learning

```

---

# Step 1 – Welcome Experience

When Pragya opens, users should see a welcoming introduction.

Example:

> Hi, I'm Pragya.

> Your AI Learning Companion.

> Let's begin your AI journey.

The welcome screen should encourage exploration rather than immediately presenting an empty chat box.

---

# Step 2 – Choose Your AI Journey

Users select one of the four strategic pillars.

Available options:

📘 Learn AI

🛠 Use AI

🛡 Stay Safe

📈 Grow with AI

This selection establishes the primary learning context.

---

# Step 3 – Choose Your Persona

After selecting a pillar, users select the persona that best represents them.

Available personas:

- Student
- Teacher
- Professional
- Parent
- Senior Citizen
- Small Business
- Just Exploring AI

Persona selection personalizes recommendations, examples, suggested questions, and future learning paths.

---

# Step 4 – Choose a Topic

After selecting the pillar and persona, Pragya presents suggested topics.

For Phase 1 MVP, suggested topics should be selected from static configuration based on:

Selected Pillar

+

Selected Persona

Example:

Journey:

Use AI

Persona:

Teacher

Suggested Topics:

- Lesson Planning
- Classroom Activities
- AI Presentation Creation
- Question Paper Generation
- Student Assessment
- Parent Communication
- AI Tools for Teachers

An "Ask Your Own Question" option should always remain available.

Users are never restricted to suggested topics.

Future versions may generate or rank topic suggestions dynamically.

---

# Step 5 – Guided Conversation

Once the learning context is established, users enter the conversation.

The conversation should use the selected context automatically.

Users should not be required to repeatedly explain their persona or learning objective.

---

# Product Decision PD-002

## Context Flexibility

**Status:** APPROVED

The Guided Learning Journey establishes the starting context only.

Users may ask any AI-related question at any point during the conversation.

Pragya should intelligently adapt to the evolving conversation while preserving relevant context.

Pragya should never force users back into the original learning path.

---

# Session Context

After the Guided Journey is completed, Pragya creates a session context.

Example:

```json
{
  "default_pillar": "USE_AI",
  "current_pillar": "USE_AI",
  "persona": "TEACHER",
  "language": "English",
  "learning_stage": "Beginner",
  "current_topic": "Lesson Planning"
}
```

This session context should be automatically included when processing future requests.

---

# Context Evolution

Pragya should continuously update the session as conversations evolve.

Example:

Initial Journey:

Use AI

↓

Teacher

↓

Lesson Planning

↓

Question:

How can AI create lesson plans?

↓

Later Question:

What are Deepfakes?

Updated Context:

Current Pillar:

Stay Safe

Persona:

Teacher

Previous Journey:

Use AI

Conversation History:

Preserved

Pragya should naturally transition between pillars without disrupting the conversation.

---

# Response Lifecycle

Every meaningful response should follow this logical sequence.

```

User Question

↓

Answer

↓

Recommended Articles

↓

Recommended AI Tools (where applicable)

↓

Continue Learning Suggestions

↓

Suggested Follow-up Questions

↓

Wait for User

```

Not every response must include every section.

The Recommendation Engine should determine which elements are appropriate.

---

# Continue Learning

Pragya should encourage continuous learning.

Instead of ending after answering a question, Pragya should recommend:

- Related articles
- Relevant AI tools
- Next learning topics
- Suggested follow-up questions
- Related learning journeys

The objective is to help users progressively improve their AI knowledge.

---

# Suggested Questions

Suggested Questions are recommendations.

They are never mandatory.

Users may ignore suggestions and ask any AI-related question.

Example:

Recommended Questions:

- How can AI create quizzes?

- Which AI tools are free for teachers?

- Can AI help weak students?

Or simply type:

> Ask anything about AI...

---

# Conversation Principles

Pragya should:

- Be helpful
- Be educational
- Be conversational
- Be encouraging
- Be simple
- Be trustworthy

Pragya should avoid:

- Technical jargon without explanation
- Long unnecessary responses
- Complex onboarding
- Forcing users into predefined paths

---

# Multi-language Experience

The Guided Journey should eventually support:

- English
- Hindi

Future versions may support additional Indian languages.

For Phase 1 MVP, English is the primary implementation language.

Hindi-ready architecture should be preserved, but full Hindi guided journey localization is not required for Sprint 1.1.

---

# End of Session

Before a conversation ends, Pragya should encourage continued learning whenever appropriate.

Examples:

- Continue this journey tomorrow.
- Explore related AI topics.
- Read recommended articles.
- Try recommended AI tools.

The goal is to transform one-time conversations into long-term AI learning journeys.

# Part 3
# Intelligence Architecture

---

# Overview

Pragya is designed as a modular intelligence platform.

Each component has a single responsibility.

This separation enables maintainability, scalability, easier testing, and independent evolution of individual components.

The architecture follows the principle:

> Understand first.
> Answer second.
> Guide continuously.

---

# High-Level Intelligence Flow

Every user interaction follows the same processing sequence.

```
User

↓

Guided Learning Context

↓

Gateway

↓

Understanding Layer

↓

Knowledge Engine

↓

Recommendation Engine

↓

Large Language Model

↓

Learning Journey Engine

↓

Response
```

Each component has clearly defined responsibilities.

---

# Guided Learning Context

The Guided Learning Context is established before the first conversation begins.

It contains:

- Selected AI Journey
- Selected Persona
- Selected Topic
- Language
- Learning Stage
- Conversation History

The context is maintained throughout the session.

---

# Gateway

## Purpose

The Gateway is the single entry point for all incoming requests.

Its responsibility is to prepare requests before they enter the intelligence pipeline.

The Gateway should remain lightweight.

It should never generate learning responses.

---

## Responsibilities

The Gateway is responsible for:

- Greeting detection
- Small talk handling
- Domain validation
- Safety validation
- Session loading
- Language detection
- Request normalization
- Request routing

---

## Responsibilities Excluded

The Gateway should NOT:

- Generate AI answers
- Retrieve articles
- Make recommendations
- Rewrite prompts
- Perform deep reasoning

Those responsibilities belong to downstream components.

---

# Understanding Layer

## Purpose

The Understanding Layer determines what the user is trying to achieve.

It interprets the request using the current session context.

---

## Responsibilities

The Understanding Layer should identify:

- User intent
- Current topic
- Relevant pillar
- Persona
- Learning stage
- Required knowledge sources
- Response strategy

---

## Context Awareness

The Understanding Layer should always consider:

- Previous conversation
- Selected pillar
- Selected persona
- Current topic
- Session history

Understanding should never rely solely on the latest user message.

---

# Knowledge Engine

## Purpose

The Knowledge Engine retrieves trusted information required to answer the user's request.

It should prioritize AINews9 knowledge before external knowledge.

---

## Knowledge Sources

Current sources include:

- AINews9 Knowledge Base
- Published AINews9 Articles
- Curated AI Learning Content
- Approved Internal Content

Future versions may support additional trusted sources.

---

## Responsibilities

The Knowledge Engine should:

- Retrieve relevant articles
- Retrieve related learning resources
- Provide supporting context
- Supply references for the LLM

---

# Recommendation Engine

## Purpose

The Recommendation Engine personalizes the learning experience.

Recommendations are generated using:

- Selected journey
- Persona
- Current topic
- Conversation history
- Learning stage

---

## Recommendation Types

Recommendations may include:

- Related Articles
- AI Tools
- Suggested Questions
- Next Learning Topics
- Related Learning Journeys

Recommendations should encourage continuous learning.

---

# Large Language Model

## Purpose

The Large Language Model generates natural language responses.

The LLM is responsible for language generation—not product behaviour.

Business rules should never depend solely on prompt engineering.

---

## Responsibilities

The LLM should:

- Produce natural responses
- Explain concepts
- Simplify complex topics
- Adapt language to the selected persona

The LLM should NOT determine:

- Product workflow
- User journey
- Recommendation strategy
- Conversation routing

Those are handled by Pragya's intelligence architecture.

---

# Learning Journey Engine

## Purpose

The Learning Journey Engine extends learning beyond a single answer.

It determines how users continue their AI learning journey.

---

## Responsibilities

The engine should:

- Recommend the next topic
- Suggest related articles
- Suggest AI tools
- Suggest follow-up questions
- Encourage continuous learning

Not every response requires all recommendation types.

The Recommendation Strategy determines the most appropriate combination.

---

# Response Lifecycle

Every meaningful response follows this logical sequence.

```
Question

↓

Understand

↓

Retrieve Knowledge

↓

Generate Answer

↓

Recommend

↓

Continue Learning

↓

Wait for User
```

The answer is one component of the complete learning experience.

---

# Context Evolution

User interests naturally evolve during conversations.

Pragya should continuously update the active context while preserving conversation history.

Example:

Default Journey:

Use AI

↓

Teacher

↓

Lesson Planning

↓

Conversation evolves to:

Stay Safe

↓

AI Deepfakes

The conversation should continue naturally without restarting the session.

---

# Design Principles

The intelligence architecture follows these principles:

- Single Responsibility
- Modular Design
- Context Awareness
- Explainability
- Maintainability
- Extensibility
- AI Model Independence

The architecture should remain stable even if the underlying LLM changes.

---

# Intelligence Principles

Pragya should:

Understand before answering.

Guide rather than simply respond.

Teach rather than merely inform.

Recommend rather than terminate conversations.

Every interaction should improve the learner's AI literacy.

# Part 4
# Technical Architecture & Engineering Requirements

---

# Overview

This section defines the technical architecture and engineering principles required to implement Pragya.

Unlike the previous sections, which describe product behaviour and intelligence, this section establishes the engineering framework that enables those capabilities.

Implementation technologies may evolve over time without changing the product behaviour defined in this specification.

---

# Architectural Principles

Pragya shall follow a modular, service-oriented architecture.

Each module should have a clearly defined responsibility.

Business logic should remain independent of:

- User Interface
- Large Language Model
- Database implementation
- External APIs
- Deployment platform

This separation enables easier maintenance, testing, and future technology changes.

---

# High-Level Technical Architecture

```
                    User
                      │
                      ▼
              Web / Mobile / API
                      │
                      ▼
                 FastAPI Server
                      │
                      ▼
               Pragya Gateway
                      │
                      ▼
             Intelligence Layer
      ┌─────────────────────────────────┐
      │ Understanding Layer             │
      │ Knowledge Engine                │
      │ Recommendation Engine           │
      │ Learning Journey Engine         │
      └─────────────────────────────────┘
                      │
                      ▼
              Large Language Model
                      │
                      ▼
                  Response
```

Future channels may include:

- WordPress Widget
- Mobile Application
- WhatsApp
- Microsoft Teams
- Slack
- Voice Assistants

All channels should use the same intelligence layer.

---

# Entry Point

The FastAPI application acts as the HTTP entry point.

Responsibilities:

- Receive requests
- Validate requests
- Authenticate (future)
- Load session
- Invoke the Gateway
- Return responses

The FastAPI layer should remain lightweight.

Business logic should not reside within API controllers.

---

# Session Management

Each user interaction belongs to a session.

A session maintains:

- Session ID
- Selected Pillar
- Selected Persona
- Current Topic
- Current Language
- Learning Stage
- Conversation History
- Active Recommendations

The session context should evolve as conversations progress.

For Phase 1 MVP, session context may be stored in memory.

This means:

- Sessions do not need to survive server restarts.
- Sessions do not need to sync across multiple backend instances.
- No authentication or user account is required.
- The implementation should keep the session model modular so persistent storage can be added later.

---

# Configuration

System behaviour should be configurable.

Examples include:

- Supported Personas
- AI Journeys
- Supported Languages
- LLM Model
- Recommendation Limits
- Search Limits
- Knowledge Sources

Business rules should be configurable rather than hardcoded wherever practical.

For Phase 1 MVP, AI journeys, personas, and topic suggestions should use static configuration.

Topic suggestions should not require LLM generation in Sprint 1.1.

The static configuration should support:

- Four AI journeys: Learn AI, Use AI, Stay Safe, Grow with AI.
- Seven persona options: Student, Teacher, Professional, Parent, Senior Citizen, Small Business, Just Exploring AI.
- Persona-and-journey-specific topic suggestions.
- A fallback set of beginner-friendly topic suggestions.

---

# Knowledge Sources

Current Knowledge Sources:

- AINews9 Knowledge Base
- Published Articles
- Internal Learning Content

Future Knowledge Sources:

- Structured Learning Modules
- AI Tool Directory
- External Trusted Sources
- User Learning History

The Knowledge Engine should prioritize trusted AINews9 content whenever possible.

---

# API Endpoints

The implementation should expose APIs that support:

## Health

Returns system health.

---

## Chat

Processes conversational requests.

---

## Recommendations

Returns personalized learning recommendations.

---

## Search

Searches relevant knowledge sources.

---

## Session

Creates, retrieves, and updates session context.

---

## Feedback (Future)

Collects user feedback to improve recommendations.

---

The API design should remain modular and versionable.

For Sprint 1.1, the API surface should support the Guided Learning Experience without overbuilding the full future platform.

Minimum implementation needs:

- Return the available journeys.
- Return personas.
- Return static topic suggestions for a selected journey and persona.
- Start or update an in-memory session context.
- Preserve compatibility with the existing chat flow where practical.

---

# Response Format

Responses should contain:

- Generated Answer
- Session Metadata
- Related Articles
- Recommended AI Tools (when applicable)
- Suggested Questions
- Continue Learning Recommendations

Not every field is mandatory.

The Recommendation Engine determines the appropriate response composition.

For Sprint 1.1 onboarding responses, the response format should prioritize predictable UI state over generated AI content.

Responses may include:

- Session metadata
- Current onboarding step
- Available options
- Selected journey
- Selected persona
- Selected topic
- Next expected action
- User-facing message

Generated AI answers are only required after the user enters the conversation state.

---

# Error Handling

The system should gracefully handle:

- Invalid requests
- Unsupported questions
- Missing knowledge
- External API failures
- Timeout conditions

Users should receive helpful messages rather than technical errors.

---

# Logging

The system should log:

- Requests
- Processing stages
- Errors
- Performance metrics

Sensitive user information should never be logged.

---

# Security

The platform should support:

- Secure API communication
- Session isolation
- Rate limiting
- Abuse prevention
- Prompt injection protection
- Input validation

Future versions may introduce authentication and user accounts.

---

# Performance Goals

Target response time:

- Greetings: <1 second
- Cached responses: <2 seconds
- Standard AI responses: <5 seconds

The system should remain responsive under normal operating conditions.

---
