# 1. Purpose

This document defines the standard Software Development Lifecycle (SDLC), branching strategy, deployment architecture, environment strategy, release workflow, and engineering practices for Pragya.

It establishes a repeatable, automated, AI-assisted engineering workflow that enables rapid feature development while maintaining production stability.

This document is the authoritative engineering reference for all Pragya software development activities.

---

# 2. Engineering Principles

Pragya follows an **AI-First Engineering** approach.

Engineering decisions shall prioritize:

- Simplicity
- Automation
- Repeatability
- Maintainability
- Low Operational Overhead
- Production Stability
- Security by Default

Core Principle:

> **Automate every repeatable engineering activity. Human effort should focus on architecture, product direction, quality and governance.**

# 4. Environment Strategy

## Development Environment

Purpose

- Feature development
- Bug fixes
- Daily coding
- Codex implementation
- Functional testing
- Integration testing

Hosting

- Render

Characteristics

- Auto-deployed from the `dev` branch
- Internal engineering environment
- May experience cold starts
- Not intended for production traffic

---

## Production Environment

Purpose

- Public users
- Stable releases
- Production traffic

Hosting

- Google Cloud Run

Characteristics

- Auto-deployed from the `main` branch
- Production-grade infrastructure
- HTTPS endpoint
- Automatic scaling
- Revision-based rollback
- High availability

# 5. Source Control Strategy

GitHub is the **Single Source of Truth**.

Rules

- All code must exist in GitHub.
- Local development must synchronize with GitHub.
- Production deployments originate only from GitHub.
- Direct production deployments from developer machines are prohibited.

- Small incremental changes are preferred.
- Render always reflects the latest development build.
- Production remains unaffected until release approval.

```
Development Complete

↓

Testing Successful

↓

Merge dev → main

↓

GitHub

↓

Cloud Build Trigger

↓

Docker Image Build

↓

Artifact Registry

↓

Deploy Cloud Run

↓

Production Validation
```

No manual production deployment should be required after the initial infrastructure setup.

---

# 9. CI/CD Strategy

The production deployment pipeline shall be fully automated.

Pipeline Responsibilities

1. Detect push to `main`
2. Build Docker image
3. Push image to Artifact Registry
4. Deploy latest revision to Cloud Run
5. Route production traffic
6. Preserve previous revisions for rollback

Developer intervention should not be required during normal deployments.

# 11. Development Platform Policy

Render is used exclusively for development activities.

Permitted

- Feature testing
- API validation
- Codex deployments
- Integration testing
- Debugging

Not Permitted

- Production traffic
- Public releases
- Long-term production hosting

---

# 12. Production Platform Policy

Google Cloud Run is the official production hosting platform.

Responsibilities

- Public API
- Stable deployments
- Automatic scaling
- HTTPS endpoint
- Revision history
- Production availability

Cloud Run shall never be used for experimental feature development.

---

# 13. Configuration Management

Application configuration shall remain external to the application.

Production configuration includes:

- Environment Variables
- Google Secret Manager
- Cloud Run Service Configuration

Rules

- No API keys in source code.
- No secrets committed to Git.
- Configuration differs by environment.
- Infrastructure manages secrets, not application code.

---

# 14. Engineering Rules

Rule 1

Never develop directly on the `main` branch.

---

Rule 2

All feature development occurs on the `dev` branch.

---

Rule 3

Only reviewed and approved code may be merged into `main`.

---

Rule 4

Production deployments originate only from GitHub.

---

Rule 5

Production deployment must remain fully automated.

---

Rule 6

Manual production deployment should be avoided.

---

Rule 7

Infrastructure modifications require architectural review.

---

Rule 8

Business logic changes must be validated on Render before production release.

---

Rule 9

Every production deployment must be reproducible.

---

Rule 10

Production configuration shall never be hardcoded.

---

# 15. Rollback Strategy

Cloud Run revision management provides native rollback capability.

Rollback Procedure

1. Select previous revision.
2. Redirect production traffic.
3. Validate production.
4. Investigate failed release.

Code rollback is not normally required.

A work item is complete only when:

- Feature implemented
- Local testing successful
- Render validation completed
- Code reviewed
- Documentation updated
- Approved for release
- Merged into `main`
- Automatically deployed to Cloud Run
- Production validated

# 19. Architecture Principles

Pragya engineering follows these core principles.

- One Repository
- Two Branches
- Two Environments
- Automated CI/CD
- Zero Manual Production Deployment
- Configuration Outside Code
- Security by Default
- Continuous Improvement

---

# 20. Long-Term Vision

The long-term objective is to create an engineering platform where software delivery is reliable, predictable, and largely automated.

The ideal release workflow is:

1. Implement feature.
2. Validate on Render.
3. Merge into `main`.
4. Automatic CI/CD deployment.
5. Validate production.

Human effort should focus on product quality and architecture rather than deployment operations.
