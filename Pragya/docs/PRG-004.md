# 1. Purpose

This document defines the implementation specification for migrating Pragya's production environment from Render to Google Cloud Run and establishing a reliable, automated CI/CD pipeline.

The migration is strictly an infrastructure initiative.

Its objectives are to improve scalability, reliability, security, deployment automation and operational maturity without changing existing business functionality.

---

# 2. Objectives

The migration shall achieve the following objectives:

- Adopt Google Cloud Run as the official production platform.
- Retain Render exclusively for development.
- Fully automate production deployments.
- Improve production reliability.
- Separate development and production environments.
- Improve security through external configuration.
- Minimize manual deployment activities.
- Preserve all existing application functionality.

# 11. Docker Requirements

The application container shall:

- Use an official Python base image.
- Install only required dependencies.
- Copy application source.
- Expose the application using Uvicorn.
- Bind to the Cloud Run PORT variable.
- Remain lightweight.
- Avoid unnecessary packages.
- Support deterministic builds.

# 16. CI/CD Requirements

The deployment pipeline shall automatically perform the following steps.

1. Detect push to `main`.
2. Trigger Cloud Build.
3. Build Docker image.
4. Push image to Artifact Registry.
5. Deploy new Cloud Run revision.
6. Route production traffic.
7. Preserve previous revisions.
8. Report deployment status.

The objective is a fully repeatable deployment process requiring minimal human intervention.

---

# 17. Rollback Strategy

Cloud Run Revision History provides built-in rollback capability.

Rollback Procedure

1. Select previous revision.
2. Redirect production traffic.
3. Validate production.
4. Investigate deployment failure.
5. Redeploy corrected version when ready.

Rollback should normally be completed within minutes.

# 19. Risks

Known Risks

- Incorrect environment configuration.
- Secret misconfiguration.
- Docker startup failure.
- Incorrect PORT binding.
- Widget configured with incorrect endpoint.
- Cloud Run scaling configuration.
- API quota limitations.

Mitigation

- Validate on Render.
- Preserve Render during migration.
- Use staged production validation.
- Verify health endpoints.
- Maintain rollback readiness.

# 23. Success Metrics

The migration will be considered successful if:

- Production deployment requires only a merge into `main`.
- Production releases are deterministic and repeatable.
- Rollback can be completed within minutes.
- Development and Production remain fully isolated.
- Deployment errors are significantly reduced.
- Infrastructure operations require minimal manual effort.

---

# 24. Engineering Principles

This migration is guided by the following principles:

- Infrastructure should be automated.
- Configuration belongs outside the application.
- Development and Production must remain isolated.
- Security should be built in from the beginning.
- Production deployments must be reproducible.
- Rollback must always be available.
- Operational simplicity is preferred over unnecessary complexity.

---

# 25. Long-Term Vision

Google Cloud Run establishes the foundation for Pragya's future platform architecture.

Future platform capabilities—including analytics, AI services, vector search, learning journeys, and personalization—should build upon this infrastructure without requiring fundamental deployment changes.

The long-term production workflow remains:

1. Develop on `dev`.
2. Validate on Render.
3. Merge into `main`.
4. Automatic Cloud Build.
5. Automatic Cloud Run deployment.
6. Validate production.

The deployment platform should become a predictable engineering utility rather than a recurring engineering task.
